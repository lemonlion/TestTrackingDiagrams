POC — 4 Bug Fixes

Issue 1: Zoom Scroll UX FIX: Drag-to-Pan

Problem

When you double-click to zoom a diagram, the container expands and a horizontal scrollbar appears at the bottom of the expanded container. To scroll right, you have to scroll all the way down to the scrollbar, scroll horizontally, then scroll back up. Very awkward.

Fix

Replace the scrollbar-only approach with drag-to-pan — click and drag anywhere on the diagram to scroll in any direction. Also constrains the container to max-height: 80vh so both scrollbars stay accessible, and switches cursor to grab/grabbing for clear affordance.

Issue 2: Floating Zoom Button FIX: prepend + visibility

Problem

The addZoomButtons() function appendChild(btn), placing it after the SVG. With position: absolute; top: 6px; left: 6px, the button should overlay, but in some layouts the position: relative on the container isn't applied, or the button element renders below the SVG.

Additionally, the button has opacity: 0.4 by default — it's always slightly visible. But the user reports it's not visible at all, suggesting either the button isn't being inserted, or the container doesn't have position: relative.

Fix

  • Use container.prepend(btn) instead of container.appendChild(btn) — ensures the button is the first child, always above the SVG
  • Set container.style.position = 'relative' before inserting — already done, but now guaranteed by prepending
  • Use CSS opacity: 0 by default and show on container hover via .container:hover .diagram-zoom-toggle { opacity: 0.4 } — cleaner than relying on individual button opacity

Issue 3: Submenu Hover Highlights All Items BUG FIX

Root Cause

The CSS rule .diagram-ctx-menu div:hover { background: #e8f0fe; } selects all descendant divs. The .submenu element is itself a <div>, so when you hover inside any submenu item, the .submenu container div also matches div:hover and gets a blue background — making it look like every item is highlighted.

Fix

Change to direct child selectors: .diagram-ctx-menu > div:hover for top-level items and .submenu > div:hover for submenu items.

Issue 4: Open Image in New Tab Creates Blank Tab BUG FIX

Root Cause

Modern browsers (Chrome 60+, Firefox, Edge) block window.open() with data: URIs as a security measure. The current code uses FileReader.readAsDataURL(blob) then window.open(reader.result) for PNG, and window.open('data:image/svg+xml;base64,...') for SVG. Both produce a blank tab.

Fix

Use URL.createObjectURL(blob) instead of data URIs. This creates a temporary blob: URL that browsers allow opening in new tabs. The "Save as" code already uses this pattern correctly.
