Zoom Toggle Investigation — Large Diagrams

This POC investigates why the zoom toggle button isn't appearing for large BrowserJs-rendered PlantUML diagrams, and proposes several fixes. Each section contains a simulated large SVG diagram to test the zoom behavior.

Current Behavior BUG

Problem

The zoom toggle (⤢) should appear in the top-left corner when hovering over a diagram, but it doesn't show for large BrowserJs-rendered diagrams. The button uses position: absolute and only shows on [data-diagram-type]:hover via CSS. The addZoomButtons() function is triggered by a MutationObserver on document.body, but there may be timing issues with the PlantUML TeaVM renderer.

Possible causes:

  • addZoomButtons() fires before SVG is inserted (hits if (!svg) return)
  • The button scrolls out of view for very tall diagrams (position: absolute; top: 6px stays at the top of the container, not the viewport)
  • The collapsible notes re-render destroys the button via container.innerHTML = ... and the MutationObserver doesn't re-fire in time
  • The CSS selector [data-diagram-type]:hover > .diagram-zoom-toggle doesn't match because the hover target is the SVG, not the container

Reproduction — Hover the diagram below to see if ⤢ appears

Option A: Sticky Zoom Button OPTION

Use position: sticky so the button stays visible as you scroll through a tall diagram. The button floats inside the container and sticks to the viewport top. Requires the container to have a scrollable or non-overflow-hidden context.

Pros: Button stays visible as you scroll a tall diagram. No JS needed for positioning.
Cons: position: sticky only works within a scrollable ancestor. If the container doesn't scroll (it's the page that scrolls), sticky won't help. The button needs to be the first element in the flow, which means it takes up space (hence negative margins).

Option B: Always-Visible Button (no hover required) OPTION

Make the zoom button always visible at low opacity (0.5) instead of requiring hover. This ensures the button is discoverable even if hover events aren't reliably detected on large containers where the mouse might be over the SVG child, not the container itself.

Pros: Always discoverable. No hover timing issues.
Cons: Adds visual noise. May overlap diagram content at top-left.

Option C: Sticky Toolbar Above Diagram RECOMMENDED

Add a sticky toolbar above each diagram that stays visible when scrolling. Contains zoom toggle, fit-to-width reset, and optionally diagram number info. This is the most robust approach — it doesn't depend on hover detection, doesn't overlap diagram content, and stays visible for diagrams of any height.

Pros: Always visible. Doesn't overlap content. Clear affordance. Can host additional controls (details radio, headers toggle).
Cons: Takes vertical space. More HTML per diagram.

Option D: Fixed-position Button (viewport-anchored) OPTION

Use position: fixed to anchor the zoom button to the viewport (not the container). The button appears when any diagram is in view, positioned at a fixed screen location. Useful for single-diagram-per-page layouts.

Pros: Always visible regardless of scroll position. Truly reliable.
Cons: Ambiguous when multiple diagrams are visible. Floats over other page content. Needs IntersectionObserver to show/hide based on which diagram is in view.
