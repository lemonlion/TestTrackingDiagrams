Prototype: Parameterized Test Grouping — Parameter Display Rules

This prototype demonstrates 4 rules for how parameters appear in the grouping table, plus diagram switching on row click.

Rules:
  1. R1 Multiple simple scalar params (≤ ~10) → one column per param, under a wide-spanning "Input Parameters" master header
  2. R2 Single complex object with simple scalar properties (≤ ~10) → flatten to columns under "Input Parameters" (same as R1)
  3. R3 Cell value is a complex item with a few scalar fields → render as a sub-table inside the cell
  4. R4 Cell value is deeply complex → expandable via <details> <summary>

Rule 1 — Multiple Simple Scalar Parameters

e.g. [Theory] [InlineData("123456", 200)] or LightBDD named params. Each scalar becomes its own column under the master header.

Rule 4 — Cell Contains Deeply Complex Object → Expandable

When a cell value is too complex for a sub-table (nested objects, arrays, large structures), it uses a <details><summary> expansion — similar to how BDD steps expand.

Decision Matrix

Parameter shape Condition Display Rule
N scalar params N ≤ ~10 One column per param under Input Parameters master header R1
Single object, all scalar props Props ≤ ~10 Flatten props to columns under Input Parameters (+ type label) R2
Cell value: small complex object ≤ ~5 scalar fields, no nesting Sub-table inside the cell R3
Cell value: deeply complex Nested objects, arrays, > 5 fields Expandable <details><summary> with preview line R4
Unstructured / custom display name Can't extract param names/values Single Test Case column with full display name text R0
All rows produce identical diagrams Single diagram + "All diagrams identical" badge; no row-click —
Row has Skip / Explicit / Not-Run status Status badge + colored row; skip/explicit rows show message instead of diagram Any
